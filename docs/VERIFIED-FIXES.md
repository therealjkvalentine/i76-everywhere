## Crashes / freezes / hangs

| Symptom | Root cause | Fix |
|---|---|---|
| Page fault at `01B82C26` at boot | Game calls `NtUserChangeDisplaySettings`; macOS refuses exclusive 640x480; game derefs null | Per-app Wine **virtual desktop**: `HKCU\Software\Wine\AppDefaults\i76.exe\Explorer â†’ Desktop=i76`, `HKCU\Software\Wine\Explorer\Desktops â†’ i76=1280x960` |

| Every Wine process pinned at **100% CPU** | `WINEESYNC=1` alone busy-spins on macOS | Set **both** `WINEESYNC=1` and `WINEMSYNC=1` (msync wins; all procs idle at ~0-4%) |

## Launch plumbing (macOS specifics)

| Symptom | Root cause | Fix |
|---|---|---|
| DxWnd GUI opens instead of the game with `/r:0` | `/R` is **1-based** (source: `iProgIndex-1`) | `dxwnd.exe /R:1` for headless one-click launch. Do **not** add `/q` â€” it suppresses the launch |
| Blank never-painting window when stub `execv`s wine | `execv` breaks winemac GUI activation under LaunchServices | Spawn wine as a **child Process** from the stub |
| .app won't launch from Finder with a script as the executable | LaunchServices refuses script bundle executables | Compile a small **Mach-O** stub |
| `codesign --deep` fails on the wrapper | Chokes on internal symlinks | Sign the stub binary alone |
| Stub's bundle sweep + relaunch detection silently never fired | `pgrep`/`pkill -f` patterns are **extended regex**; the stub passed the raw app path, whose `(DxWnd)` parens are metacharacters â†’ matched nothing (cleanup happened to survive on `wineserver -k` alone) | Stub escapes literal paths via `NSRegularExpression.escapedPattern` before pattern use (`rx()` in [i76-launch-stub.swift](../i76-launch-stub.swift), fixed 2026-07-14) |
| macOS asks for **microphone permission** at launch | The game/Wine opens a capture stream at audio init (winecoreaudio starts every capture stream at creation â€” `AudioOutputUnitStart` â€” which is what TCC gates; enumeration alone doesn't prompt). **No Wine registry key/env/DllOverride exists to disable capture while keeping output** (verified against Wine 10 source; the only knob, `Drivers\Audio=""`, kills ALL audio) | **Deny once â€” it's permanent** (TCC stores it per bundle ID; denied capture just reads silence, output unaffected). It re-prompts once after the wrapper binary is re-signed (our stub rebuilds do that) â€” deny again |

## Audio / music

| Symptom | Root cause | Fix |
|---|---|---|
| Music in cutscenes but **silent in missions** | Mission music = CD **redbook audio**; GOG ships it as `music/N.mp3` + an *empty* `tracklen.nfo`; no CD to play under Wine | DxWnd **virtual CD audio** (`flagm0` bit 0 â†’ dxwplay.dll) + hard-link tracks to its expected `Music\TrackNN.mp3` naming + delete the empty `tracklen.nfo` (it regenerates). | **Wrong music over FMV cutscenes** (mission/office track bleeds over the movie's own chill score) | **Known GOG bug, NOT the port.** The 29 `smk/*.SMK` cutscenes carry their own baked audio (voice + Salazar score, Smacker audio track 1 â€” verified in the headers). GOG's file-based music (our virtual CD) has no such limit, so the last track keeps playing over the movie. Happens identically on GOG-for-Windows; the AiO patch doesn't fix it either. [GOG thread](https://www.gog.com/forum/interstate_series/i76_music_plays_during_cutscenes) | **FIXED â€” our `SMACKW32.DLL` proxy** ([smack-music-fix/](../smack-music-fix/), install: [setup-cutscene-music-fix.sh](../setup-cutscene-music-fix.sh)). Mission music untouched (the game starts each track itself). Fallbacks if ever needed: `setup-music.sh --off` (silence CD music entirely) or Abortâ†’Replay. Revert: `setup-cutscene-music-fix.sh --revert`. **Ear-confirmed in-game 2026-07-14.** Known cosmetic residue: ~1 s of the old track over the "somewhere in the southwest"-style mission fade-ins â€” that's the engine's own track sequencing on IN-ENGINE cutscenes (the proxy can only fire at `SmackOpen`, i.e. FMV starts; the game calls it after the fade begins). No cheap fix; accepted |

## Performance / visuals (measured ceilings)

| Fact | Detail |
|---|---|
| **20 FPS cap ships in the GOG exe** | GOG 2019 `i76.exe` is byte-identical to UCyborg's AiO patch; `I76PATCH.DLL` = built-in 20 FPS physics limiter. **Measured ~20.66 FPS in-sim** â€” physics-safe with no external limiter. So the DxWnd limiter below only *bites* when set **under ~20.66** |
| **DxWnd FPS limiter: how to actually set it (was silently off)** | Two parts, BOTH required â€” the value alone does nothing: **(1) enable** = `flagg0` bit **`0x1000`** (LIMITFPS, the "Limit frame rate" checkbox); **(2) value** `maxfps0` = **delay in ms** when the "Hz" flag (`0x4000`, LIMITFREQUENCY) is **off** (our mode), with `FPS = 1000/ms`. So **50 ms = 20 FPS, 51 ms â‰ˆ 19.6, 52 ms â‰ˆ 19.2** (whole-ms granularity â€” 19.5 exactly isn't expressible). Decoded from DxWnd's own example profiles + `help/Timing.html`. Current shipped: **`maxfps0=52` (â‰ˆ19.2 FPS)**, `flagg0=1543507984` â€” set deliberately a touch under 20 so the Mission 5 canyon jump *and* the tighter later-mission bridge gaps (tuned for the era's exact-20 sim) clear reliably. Both the live `drive_c/dxwnd/dxwnd.ini` (profile 0) and the repo `interstate-76.dxw` must be edited (CRLF files â€” anchor edits at line start) |
| **1024x768 = software renderer max** | Menu maxes there because the engine does; no config/registry/hex unlock found; D3D mode is locked 640x480. DxWnd upscales output beyond it |
| **Default is 640x480; set 1024x768 in-game** | The software renderer's resolution lives in `I76PLYR.DEF` (binary player-prefs, no readable dimensions â€” it's an index byte). Set it via **Options â†’ Graphic Detail â†’ Screen Resolution â†’ 1024x768**; it persists (the close-fix now saves cleanly on quit). Can't be baked as a shipped default headlessly: the 2D menu renders **black to `screencapture`** under DxWnd's primary-surface renderer (3D captures fine, menu doesn't), so no before/after diff to locate the byte. To bake it: set it once, then diff `I76PLYR.DEF` vs a 640x480 baseline |
| Camera is hardcoded 4:3 | No widescreen exists; "fill screen" = big 4:3 + black bars |
| Higher *internal* res needs Glide | dgVoodoo 2.78.2 (Glideâ†’D3D11â†’DXVKâ†’MoltenVK) renders 2x+ with 3dfx gamma â€” the "Voodoo" launcher. | **HD pack: night missions (e.g. Mission 6) color-shifted; day worlds fine** | The pack ships night only as software **VQM** (`nightm.pak`), and VQM is indexed against each level's own 8-bit `.ACT` palette (no per-tile palette like Glide M16) â€” the pack's night VQM was quantized against the wrong palette. ## Gameplay quirks (engine, not the port)

| Symptom | Root cause | Fix |
|---|---|---|
| **Mission 5 canyon jump falls just short** (no other way out) | I76's jump distance is **inversely tied to framerate** - above 20 FPS the car falls too fast per frame and lands short ([Local Ditch](https://www.localditch.com/posts/fps-jumping/), [GOG forum](https://www.gog.com/forum/interstate_series/argh_help)). The GOG exe's AiO limiter targets 20 but overshoots to ~**20.66** here - just enough to miss | Cap to exactly 20: DxWnd Timing tab: check **Limit** and set **delay (msec) = 50** (the field is *ms-per-frame* = 1000/FPS, so 20 FPS = 50 ms â€” NOT `maxfps0=20`, which is 50 FPS!). Enabling the Limit checkbox is required; the value alone does nothing. Throttles the ddraw Flip -> the game loop incl. physics runs at 20. Then **use nitrous** on the ramp approach for margin |
| **Nitrous equipped but nothing happens** | Nitrous is a **mounted "special"**, not a "nitrous key" - `nitrous_on`/`nitrous_off` are internal engine states, NOT bindable input actions (confirmed: they sit in a different exe string table from the real actions). Activated via a **special slot** (`special1/2/3`) | Bind all three special slots to keys (this build left `special1` on a joystick button only). We set `special1=6` (the stock default), `special2=7`, `special3=8`; nitrous fires from whichever slot it's mounted in. See [I76-GAMEPLAY-REFERENCE.md](I76-GAMEPLAY-REFERENCE.md) |
